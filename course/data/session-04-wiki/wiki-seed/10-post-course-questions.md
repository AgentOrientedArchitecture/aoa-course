# Post-Course AOA Questions

This note gives the Session 4 wiki a small set of practical answers to common
post-course questions. It is intended to make the demo wiki useful after the
course, not just during the live build.

## How do we get started with AOA?

Start with one business domain or value stream. Inventory the AI tools,
automations, agents, copilots, prototypes, and vendor features already in that
boundary. Identify owners, data touched, tools used, autonomy level, risk tier,
human oversight, lifecycle state, and evidence.

Then choose one or two candidate Agentic Units. Write a lightweight inventory
record for each. Score the ten maturity dimensions separately. Use the scores
to find the bottleneck capability that blocks safe progress. Build a Now /
Next / Later roadmap from that evidence.

Do not start by building an enterprise agent registry unless discovery and
reuse are already the bottleneck.

## How do we create context-aware principles?

Run a principles workshop after the initial inventory. Use the actual
inventory to identify non-negotiables, autonomy boundaries, evidence
standards, interoperability rules, registry criteria, fitness criteria, and
review cadence.

The output should not be a generic values poster. It should be a principles
map with local rules, owners, examples, and evidence requirements.

## When should we build a registry?

Build a registry when there are enough useful candidate AUs that discovery,
reuse, ownership, lifecycle state, and selection become real problems.

Before that, build the inventory and candidate AU list. A registry without
owned, evaluated, lifecycle-managed capability records becomes a static phone
book.

## Where do A2A and MCP fit?

Use MCP-style interfaces for tool and data access when they reduce bespoke
connectors and can be governed. Use A2A-style interfaces for agent-to-agent
task delegation, discovery, and long-running collaboration.

Do not treat either protocol as the whole architecture. AOA still needs
ownership, capability cards, registry semantics, evaluation evidence, runtime
control, human oversight, and observability.

## What should be an Agentic Unit?

A good AU candidate performs one meaningful capability with a stable boundary.
It has named ownership, clear inputs and outputs, declared tools or systems,
known autonomy, risk tier, evaluation evidence, and a reason to be discovered
or reused.

Avoid turning every prompt, script, or vendor feature into an AU. Some things
should stay local until their boundary, value, and evidence are clearer.

## How do we avoid maturity-model theatre?

Do not ask "what maturity level are we?" Ask "which dimensions are mature
enough for the next adoption move in this domain?"

Score the dimensions separately and require evidence. A single averaged score
hides the weakness that will break the roadmap, such as good experimentation
with weak governance or good standards adoption with poor observability.

## What should we do in the first week?

1. Pick one domain or value stream.
2. Create an AI and agent inventory.
3. Select one candidate AU.
4. Write the lightweight AU inventory record.
5. Score the ten maturity dimensions.
6. Run a short principles workshop.
7. Draft a Now / Next / Later roadmap.
8. Agree the first governance review.
